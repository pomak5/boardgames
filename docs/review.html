Проект значительно выше среднего уровня для MVP/PoC. Аккуратная монорепо-структура, единый чистый движок без I/O, строгий TypeScript, типизированные socket-события, хорошее тестовое покрытие Codenames и продвинутый бот-капитан на эмбеддингах. Основные проблемы сосредоточены в инфраструктуре вокруг кода: нет закоммиченных миграций Prisma, нет кеширования в CI, ненадёжное хранение состояния комнат (только в RAM), и одно детерминизм-нарушение в движке Uno. Архитектурные принципы — выдержаны

GOODСервер авторитарен, единый движок

packages/shared — единственный источник игровой логики. Сервер валидирует ходы, клиент шлёт намерения. Пер-view-редакция состояния сервером (redactCodenames, redUno) — корректно прячет чужие руки/ключ. Это правильно и безопасно.

GOODРеестр игр — точка расширения

apps/server/src/games.ts:25 — массив GameModule[]. Чтобы добавить игру, нужна одна запись; index.ts:33 поднимает неймспейс автоматически. Чистый plugin-паттерн.

GOODДвижок чистый (no I/O)

Все функции движка — чистые трансформеры состояния. Нет сети, таймеров, Date.now(), логирования. Рандом инжектируется параметром (codenames/engine.ts:24), что делает их детерминированными и тестируемыми. Uno с 2026-06-17 тоже инжектирует random (CRITICAL-фикс закрыт, см. трекер).

MEDIUMСостояние комнат только в RAM — теряется при рестарте

Оба менеджера хранят комнаты в private rooms = new Map<...> (codenames/manager.ts:66, uno/manager.ts:95). При падении/рестарте сервера все активные партии исчезают, игроки не смогут rejoin. Для MVP с self-hosted VPS допустимо, но это главный архитектурный риск для надёжности.

Для прод-нагрузки — периодический снапшот активных комнат в Redis/Postgres (раз в N секунд или на смену фазы), либо явное признание «партия живёт до рестарта» в SLA. Снапшотить только невидимую часть (своё состояние), видимое уже приходит по сокету.
MEDIUMapps/web намеренно вне workspaces — две системы сборки

Web — standalone, форматируется Biome; корень — Prettier + ESLint. Решение задокументировано в CLAUDE.md:124 и сделано чтобы обойти frozen-lockfile конфликт. Но это удваивает когнитивную нагрузку и инфра-настройку, и источник постоянных «почему CI падает»-ситуаций (как с bun.lock в этом ревью).

Если frozen-lockfile был единственной причиной выноса — рассмотрите возврат web в workspaces с единым lockfile; bun workspaces это поддерживают. Иначе держать, но явно задокументировать как технический долг.

GOODБот-капитан Codenames — высокого качества

Cosine-сходство над int8-квантованными navec-эмбеддингами с корректной денормализацией (codenames/bot.ts:52). Многокритериальный скоринг: размер группы → margin targetSim − dangerSim. Жёсткий потолок по киллеру ASSASSIN_CAP = 0.35 (bot.ts:28,120). Взвешенные опасности (киллер 1.0, оппонент 0.85, нейтрал 0.6). Профили риска реально меняют порог. Тесты на 50 сидах.

CRITICALUno нарушает детерминизм — Math.random вшит в движок uno/deck.ts:27 · uno/round.ts:43 · uno/bot.ts:36

shuffle() (deck.ts:27) и createUnoRound (round.ts:43) хардкодят Math.random(), а бот (bot.ts:36) решает челлендж через Math.random() < p. Codenames для сравнения всю случайность прокидывает параметром (engine.ts:24). Следствия:

  • Невозможны детерминированные тесты Uno (создание раунда, поведение бота на челлендже).
  • Нельзя реплеить/отлаживать партию — каждое создание колоды непредсказуемо.
  • Ломает чистый контракт «движок = чистая функция».
Прокинуть random: () => number параметром во все функции Uno (как сделано в Codenames). В проде передавать Math.random, в тестах — seeded LCG. Заодно это позволит добавить property-тесты на полную партию Uno.

✅ Закрыто (2026-06-17): random прокинут в shuffle, createUnoRound и botAction (поле UnoState.random); в проде Math.random, в тестах — seeded LCG. Тесты «детерминизм (random injection)» зелёные. См. трекер и CLAUDE.md (веха 2026-06-17).

MEDIUMДублирующая запись в лог при игре wild-карты uno/play.ts:117 + uno/special.ts:11

При розыгрыше wild: playCard пишет log('play') (play.ts:117), затем фаза → chooseColor, и chooseColor пишет log('play') ещё раз (special.ts:11). Клиенты видят log.slice(-30) — получат дубликат. Не сломает игру, но «оском» в истории хода.

В playCard для wild-карты не писать лог (или писать «ожидаю выбора цвета»), а финальный play-лог оставлять только в chooseColor. Покрыть тестом.

✅ Закрыто (2026-06-19): playCard не логирует wild/wild4 (кроме выигрышной карты — там chooseColor не зовётся), chooseColor пишет один play с выбранным цветом. +2 теста-регрессии в engine.test.ts.

MEDIUM1.2 МБ эмбеддингов тащатся в любой бандл импорта shared codenames/bot.ts:7

bot.ts статически импортирует embeddings.data.json (1.2 МБ). Так как index.ts реэкспортит ./codenames → ./bot, любой web-бандл, импортящий хоть что-то из @boardgames/shared, тянет весь блоб — даже Uno-only клиенты или клиенты, которым нужны только типы. Ленивая загрузка (load() при первом вызове) есть, но JSON-строка уже в бандле.

Вынести bot + embeddings в отдельный entry (@boardgames/shared/bot) или динамический import(). Сервер импортит напрямую, web — только если реально показывает бота.

LOWInfinity в guessesLeft не JSON-сейф codenames/engine.ts:101 → view.ts:16

count===0 даёт guessesLeft = Infinity, в view перекодируется в 'unlimited'. Нагрузка на Number.isFinite в view. Если состояние когда-нибудь будут сериализовать в БД — Infinity превратится в null и сломает логику.

⏳ Отложено к persist (§1): проверено — CodenamesState нигде не сериализуется в JSON (RAM-only; по проводу идёт redactCodenames, который уже конвертирует Infinity→'unlimited' через Number.isFinite). Реальная экспозиция возникнет только при persist+restore (MED §1) — тогда null после round-trип сломал бы guess() (null-1=NaN, NaN>0=false → досрочный endTurn). Фикс (sentinel вместо Infinity + guard в guess()) скоординирован с задачей persist, чтобы не трогать engine-семантику без нужды и без e2e-покрытия.

4Сервер · socket.io + Fastify

Чистая слоистая архитектура: bootstrap → реестр → неймспейсы → manager + handlers. Хорошо структурированные таймеры и планирование ботов.

GOODПаттерн guard + broadcast единообразен

Все хендлеры обёрнуты в guard() (codenames/handlers.ts:231), который ловит RoomError → game:error, а прочее → логирует. broadcast() обходит сокеты комнаты и шлёт каждому его персональный view. Согласованно в обеих играх.

GOODТаймеры Codenames продуманы

Двойной таймер: firstTurnSec на первый ход, при просрочке капитана — «время игроков» (grace), и только потом ход переходит (handlers.ts:168-200). Бонус bonusSec за верное слово (addGuessBonus). Снятие таймера при любой смене состояния. Условная логика «ждём живого капитана» — корректна.

HIGHТаймеры хранятся в setTimeout без валидации на рестарте codenames/manager.ts:51 · uno/manager.ts:52

Каждая комната хранит timer: ReturnType<typeof setTimeout>. При рестарте процесса все таймеры теряются → ход может зависнуть навсегда (никакой fallback-логики «комната простояла N минут → закрыть»). В сочетании с RAM-only состоянием (§1) — это означает зависшие/зомби-комнаты после любого деплоя.

Либо persist + восстановление таймеров при старте, либо janitor-задача: раз в минуту вычищать комнаты без живых сокетов дольше порога. Дедлайн уже хранится (turnDeadline) — достаточно на реконнекте перепланировать таймер по нему.
MEDIUMrecordFinish гонка при нескольких сокетах одного юзера codenames/handlers.ts:63 · uno/handlers.ts:44

Сбор recipients идёт по nsp.sockets. Если у одного userId открыто 2 вкладки — он попадёт в recipients дважды и результат запишется дважды (2 строки GameResult). Дедуп по userId перед записью (new Map(userId → team)), либо уникальный индекс БД на (userId, finishedAt-окно). Перенести флаг recorded из handler-scope в поле комнаты.

Идентичные функции получения ника/аватара по userId. Нарушает DRY; при правке одной вторая рассинхронизируется.

Везде void socket.join(room.code). join в socket.io возвращает Promise (при использовании Redis-адаптера). В single-node — безопасно, но при горизонтальном масштабировании room-join может не успеть — и broadcast через nsp.to() не дойдёт. Сейчас broadcast идёт по nsp.sockets перебором, так что не критично, но ломает будущий multi-node.

⏳ Отложено к Redis-адаптеру (§10): на single-node (in-memory adapter) socket.join синхронный — await не добавляет ценности. Фикс (10 сайтов + 5 async-конверсий в 3 хендлерах) скоординирован с подключением Redis-адаптера (MED §10), где он реально нужен и будет проверен вместе. Сейчас — churn без e2e-покрытия и verifiable benefit.

5Безопасность

Авторизация реализована корректно для MVP. Несколько реальных рисков, ни одного критического.

GOODПароли — argon2id, JWT — HS256 через jose

Bun.password.hash (по умолчанию argon2id) — современно. JWT через проверенную библиотеку jose, HS256, срок 30 дней. Guard в index.ts:15-20: если задан DATABASE_URL, но нет JWT_SECRET — сервер не стартует. Прод не загрузится с dev-секретом.

GOODPer-player redaction состояния

Сервер рассылает каждому сокету его персональный view (redactCodenames/redactUno). Чужие руки/ключ не утекают по проводу. Это правильная модель для авторитарного сервера.

GOODRate-limiter на auth-роутах

In-memory лимитер 10 попыток/мин по IP+маршрут (routes.ts:35-51) на register/login. Защита от брутфорса и спам-регистрации. Самоочистка при >5000 записей.

HIGHНет валидации входных данных сервером (schema validation)

REST-роуты (routes.ts) и socket-хендлеры читают req.body / аргументы как as RegisterBody без runtime-валидации. Никакого zod/ajv/Fastify-schema. В коде уже есть zod ^4 в web-зависимостях. В хендлерах — полностью доверие к TS-типам от клиента. Любой malformed payload (число вместо строки, огромный cardIndex, чужой targetId) пройдёт до движка.

Движок во многом защищён (валидация внутри), но нужен периметр: Fastify schema validation на REST + zod-схемы на socket-аргументы. Особенно room:settings, game:act, chat:send (текст до 500 — но проверяется ли длина в обработчикеUno? да, manager.ts:283 ok). Минимум — тип + длина на все строковые поля.
HIGHCORS разрешает только один origin из env — но дефолт localhost index.ts:9,12,30

WEB_ORIGIN ?? 'http://localhost:5173'. Если при деплое забыть WEB_ORIGIN — сокеты/REST будут принимать только localhost, что тихо сломает прод. Это «fail-closed», что хорошо, но неочевидно для оператора.

Либо требовать WEB_ORIGIN явно в проде (как JWT_SECRET), либо warning в логе при дефолте. Документировать в deploy-гайде.
MEDIUMJWT в localStorage — уязвим к XSS

Токен в localStorage (auth.ts:5). Любой XSS утянет токен. HttpOnly-cookie был бы безопаснее, но сложнее с cross-origin (web :5173 ↔ server :3001). Для MVP приемлемо, но это технический долг: CSP-заголовки и sanitize пользовательского ввода (никнейм, чат) становятся обязательными.

MEDIUMНикнейм/чат выводятся без явной санитизации

Никнеймы и сообщения чата проходят через trim+slice, но проверка, что они НЕ содержат разметку/скрипт, зависит от того, как фронт их рендерит. Если где-то dangerouslySetInnerHTML или небезопасный атрибут — XSS. Нужен аудит React-рендера + Content-Security-Policy заголовок от сервера статики.

✅ Закрыто (2026-06-19, рендер): аудит apps/web/src — dangerouslySetInnerHTML нигде нет; никнеймы/текст чата рендерятся как JSX text ({m.text}, {p.nickname}) и в aria-label/alt — React auto-экранирует; style-атрибуты используют статичные CSS-var; <img src={avatarUrl}> не исполняет javascript:, а avatarUrl валидируется сервером (data:image/ + лимит); NAV_LINKS статичные. XSS-вектор через рендер отсутствует.

⏳ Отложено (CSP-заголовок): meaningful CSP требует server-side header (env-aware connect-src для dev :3001 vs prod same-origin + nonces для prod-ассетов). Meta-CSP либо ломает Vite-dev (inline HMR), либо с 'unsafe-inline' теряет смысл. Static-сервер SPA не в репо — CSP добавляется при deploy-настройке (nginx/Vite preview).

MEDIUMАватары как data-URL — потенциал для;base64-атак

Хранение data:image/* в БД (routes.ts:97-109) валидируется префиксом и длиной (140 КБ). Но нет проверки magic bytes/реального декодирования — злоумышленник может заслать data:image/png, + произвольный payload до лимита. Само по себе не RCE, но базу можно забивать мусором. И каждое чтение юзера тянет ~140 КБ.
